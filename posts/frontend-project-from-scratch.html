Frontend React Architecture with OpenAPI and Microservices

Overview

This document describes a practical frontend architecture for React applications when backend microservices and OpenAPI specifications already exist.

The architecture focuses on scalability, typed API integration, CI/CD readiness, Kubernetes deployment, and enterprise-grade maintainability.

1. Goals

  • Frontend typed completely using TypeScript
  • Automatically generate API clients from OpenAPI
  • Reduce handwritten API logic
  • Synchronize frontend/backend contracts
  • Support scalable microservice architectures
  • Be compatible with CI/CD and Kubernetes deployment

5. Frontend Principles

5.1 Frontend must go through API Gateway

GOOD
http://gateway:8080/identity/users

BAD
http://identity-service:8081/users

Benefits:

  • Centralized authentication
  • Centralized logging
  • Centralized routing
  • Easier Kubernetes scaling
  • Better security control

5.2 Do not handwrite API clients

API clients should be generated directly from OpenAPI specifications.

Recommended tool: Orval

9. HTTP Layer

9.1 Axios Instance

import axios from 'axios';

export const http = axios.create({
  baseURL: import.meta.env.VITE_API_URL,
  timeout: 30000,
});

9.2 JWT Request Interceptor

http.interceptors.request.use((config) => {
  const token = localStorage.getItem('access_token');

  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }

  return config;
});

9.3 Error Interceptor

http.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.response?.status === 401) {
      localStorage.clear();
      window.location.href = '/login';
    }

    return Promise.reject(error);
  }
);

10. Orval Mutator

import { http } from './http';

export const customInstance = async <T>(
  config: any
): Promise<T> => {
  const response = await http(config);

  return response.data;
};

Update Orval Configuration

override: {
  mutator: {
    path: './src/lib/orval.ts',
    name: 'customInstance',
  },
},

11. Authentication Architecture

Authentication Flow

React
   -> Keycloak Login
       -> access_token
           -> API Gateway
               -> downstream services

Detailed Login Flow

1. User opens frontend
2. Redirect to Keycloak
3. User logs in
4. Receive access token
5. Store token
6. Attach Bearer token to API requests
7. Gateway validates JWT
8. Gateway forwards request to services

20. Best Practices

20.1 Commit OpenAPI JSON into Git

  • Track API changes
  • Support reproducible builds
  • Stabilize CI/CD pipelines

20.2 Commit Generated API Code

  • Frontend builds become independent from backend runtime
  • CI pipelines become more stable
  • Debugging becomes easier

20.3 Backend must define operationId clearly

@Operation(operationId = "getUsers")

Avoid generated names like:

usersControllerGet

20.4 Do not expose Entity objects directly

Backend services should expose DTOs instead of JPA entities.

22. Conclusion

This frontend architecture is designed for enterprise microservice systems running on Kubernetes with API Gateway and centralized authentication.

The architecture is fully typed, scalable, CI/CD friendly, and optimized for long-term maintainability.

Particularly suitable for:
  • Spring Boot microservices
  • API Gateway architectures
  • Keycloak authentication
  • Kubernetes platforms
  • Enterprise React applications
